# Day6-问题

主题：为什么 retrieval 的本质是“在回答前拿对上下文”

## 1. 常见问题

### Q1. retrieval 和 RAG 是一回事吗？

参考作答方向：

- retrieval 是获取外部上下文
- RAG 是把 retrieval 和 generation 结合起来
- retrieval 是基础，RAG 是完整应用模式

### Q2. 为什么 2-step RAG 常常更适合入门和很多实际场景？

参考作答方向：

- 流程更稳定
- 调试更简单
- 成本和延迟更可预测
- 对简单问答场景非常实用

### Q3. 为什么知识库不等于“把文档放进文件夹”？

参考作答方向：

- 文档必须被加载、切分、向量化、存储，才能被有效检索
- 真正重要的是“能不能在问题到来时拿到相关内容”

## 2. 面试问题

### Q1. 2-step RAG 和 agentic RAG 的核心区别是什么？

作答要点：

- 2-step RAG 先检索再生成，流程固定
- agentic RAG 由 agent 决定何时检索、如何检索
- 前者更可控，后者更灵活但复杂度更高

### Q2. 为什么说 retrieval 解决的是“上下文获取”问题，而不是“模型能力增强”问题？

作答要点：

- 模型本身没有变
- 变化的是回答前拿到了相关外部信息
- 本质上是在补运行时上下文

### Q3. 一个最小 RAG 系统通常包含哪些关键组件？

作答要点：

- 文档加载
- 切分
- embeddings
- vector store
- retriever
- 生成模型

## 3. 开脑问题

### Q1. 如果检索到的文档不相关，RAG 会变好吗还是变差？

思考方向：

- retrieval 不是天然增益
- 检错上下文会误导生成
- 这说明“检索质量”比“有没有检索”更重要

### Q2. 为什么很多人做 RAG 时只关注模型，却忽略切分策略？

思考方向：

- 因为模型更显眼
- 但 chunk 太大或太小都会直接影响检索质量

### Q3. 如果知识库里根本没有答案，系统应该怎么表现才算好？

思考方向：

- 应该诚实说明上下文不足
- 不应该因为做了 RAG 就强行编造

## 4. 今日输出要求

请你在完成代码后，至少自己回答下面两个问题：

1. 为什么说 retrieval 本质上是在做“运行时上下文补全”，而不是“给模型外挂知识大脑”？
- 模型本身没有变
- 变化的是回答前拿到了相关外部信息
- 本质上是在补运行时上下文

2. 如果一个问题无 RAG 时回答得更顺，但有 RAG 时反而更差，你会优先怀疑哪一层？为什么？
- 我会优先怀疑 retrieval 链路，而不是先怀疑模型本身
- 首先看 retriever 拿回来的文档是否相关，因为错误上下文会直接误导生成
- 其次看 chunk 切分是否合理，过大或过小都会影响检索质量
- 再看 embeddings 和 vector store 配置是否合适
- 最后看 prompt 是否把检索结果组织清楚；如果把低质量上下文硬塞给模型，有 RAG 反而会比无 RAG 更差
