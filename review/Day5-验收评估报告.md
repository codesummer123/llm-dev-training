## 2. 做得好的点

### 2.1 已经真正抓住了 middleware 的两个核心方向

这次不是“写了两个装饰器”这么简单，而是分别命中了两类典型控制能力：

- `wrap_model_call`：做动态 prompt 注入
- `before_model`：做调用前短路拦截

这正是 middleware 最值得掌握的两条主线。

### 2.2 能区分 transient 修改和 persistent 状态

这是 Day5 最重要的抽象之一。  
你已经能比较清楚地理解：

- 动态 prompt 更适合当前调用的瞬时增强
- 不应该无脑写回 state

这个认知非常关键，因为后面学 memory、middleware、LangGraph state 时，很多混乱都来自这里没有分清。

### 2.3 愿意修关键行为 bug，而不是只看测试绿不绿

这次最有价值的一点是：

- 测试一开始其实是绿的
- 但你愿意继续修正 `get_latest_user_text()` 取错消息的问题

这说明你已经越来越像在做真实工程，而不是只在“刷通过”。

### 2.4 问答理解进入了更成熟的层级

Day5 的两个核心问题回答已经比较稳：

- middleware 是运行时控制面
- 动态 prompt 如果持久写入 state，会污染后续上下文

这说明你已经开始从“技巧视角”转向“系统视角”。

## 3. 本轮关键修正

### 3.1 修复了 latest user 读取逻辑

最初 `get_latest_user_text()` 是从前往后读取消息，导致：

- 取到的是最早的用户输入
- 动态 prompt 和过宽请求判断都有可能基于旧上下文做出错误决策

这次修成倒序读取后，Day5 的 middleware 行为才真正对齐了训练目标。

这个修正很关键，因为它不是样式问题，而是运行时行为正确性问题。

Day5 是一个重要转折点。

而是开始把它当成一个带控制面的系统。

## 6. 仍可继续优化的点

### 6.1 当前测试主要覆盖辅助函数，还没有直接验证 middleware 的端到端行为

现在的测试足够支撑 Day5 训练目标，但未来可以继续补：

- 动态 prompt 注入是否实际发生
- 过宽请求是否真的在模型调用前短路
- 不同输入是否导致不同风格回答

### 6.2 过宽请求判断目前还是基于启发式规则

这在 Day5 非常合理。  
但也要意识到：

- 启发式规则容易漏判或误判
- 后面可以进一步加入更细致的策略组合

## 7. 一句话总结

Day5 的真正成果，不是“会写 middleware”，而是开始建立：

`把 Agent 看成一个可在运行时被塑形、被限制、被控制的系统。`
