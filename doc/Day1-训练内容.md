# Day1-训练内容

主题：LangChain 第一性原理与最小工具型 Agent

## 1. 学习目标

今天的目标不是“把一个 demo 跑起来”，而是吃透 LangChain Agent 最底层的三个核心对象：

- `Model`
- `Message`
- `Tool`

完成今天后，你应该能回答：

- LangChain Agent 到底是什么
- 它为什么不只是一次普通的模型调用
- 一个最小 Agent 的代码骨架应该怎么写

## 2. What

### 2.1 今天要学的知识点

- `create_agent` 的角色
- chat model 的初始化
- message 的基本形态
- tool 的定义方式
- system prompt 的作用
- agent `invoke` 的输入输出结构
- “模型推理 + 工具调用 + 最终回答”的基本循环

### 2.2 核心概念

#### Model

模型是推理引擎，负责理解输入、决定是否调用工具、组织最终输出。

#### Message

消息是模型的上下文载体。至少要理解：

- system message
- user message
- assistant message
- tool message

#### Tool

工具是 Agent 与外部世界交互的能力接口。  
工具本质上是“给模型暴露出的、可被调用的函数”。

#### Agent

Agent 不是“模型会说话”这么简单，而是一个循环系统：

1. 读取上下文
2. 让模型判断是否需要工具
3. 执行工具
4. 把工具结果再喂给模型
5. 直到模型给出最终回答

## 3. Why

### 3.1 为什么今天只学这几个点

因为后面 LangGraph、Deep Agents 的大部分复杂能力，最终都建立在这几个最小抽象之上。

如果今天这三个问题没吃透，后面会越来越乱：

- 模型到底负责什么
- 工具到底负责什么
- 状态和上下文为什么会影响决策

### 3.2 为什么很多人学了 Agent 还是不会做系统

因为他们学的是“怎么跑示例”，不是“这个系统为什么这样设计”。

今天要建立的第一性认知是：

- 模型负责推理，不负责真实执行
- 工具负责执行，不负责理解任务全貌
- prompt 决定行为边界
- message 决定上下文质量

很多 agent 表现差，不是模型不行，而是：

- prompt 不清
- tool schema 不清
- message 组织差
- 给模型的上下文噪声太大

## 4. How

### 4.1 今天的训练任务

你要亲手实现一个最小的“学习助手 Agent”，它具备三种能力：

- 解释一个框架概念
- 给出一个学习动作建议
- 估算一个练习任务需要的时间

注意：今天所有工具都用本地 mock 数据完成，不依赖数据库、不依赖网络、不依赖中间件。  
目的不是做业务，而是把 Agent 核心骨架写对。

### 4.2 代码任务拆解

#### 任务 1：初始化模型

使用官方推荐的 `init_chat_model` 初始化一个支持 tool calling 的 chat model。

要求：

- 模型名从环境变量读取
- 不把 provider 写死在代码逻辑里
- 为后续切换模型保留空间

#### 任务 2：定义三个工具

建议实现：

- `lookup_concept`
- `suggest_next_step`
- `estimate_study_time`

要求：

- 使用 `@tool`
- 给每个工具写清晰描述
- 返回值尽量稳定、短小、可预测

#### 任务 3：写 system prompt

system prompt 约束这个 Agent：

- 优先使用工具，不乱编造
- 对不确定问题要诚实
- 输出要简洁、结构清楚

#### 任务 4：创建 Agent

用 `create_agent` 组合：

- model
- tools
- system prompt

#### 任务 5：完成最小调用

至少准备两组输入：

- 一组不需要工具
- 一组明显需要工具

观察：

- 输出内容
- 工具是否被调用
- 最终回答是否引用了工具结果

### 4.3 推荐运行流程

1. 先只让模型回答一个普通问题
2. 再挂一个工具
3. 再挂三个工具
4. 最后调 prompt

不要一口气把所有东西堆上去，否则你会分不清问题出在哪一层。

## 5. 代码设计说明

今天的代码骨架已经放到 `code/Day1/`。

建议职责划分：

- `main.py`
  - 构建模型
  - 构建 agent
  - 提供一个最小运行入口
- `tools.py`
  - 定义 3 个工具
- `prompts.py`
  - 放 system prompt
- `tests/test_smoke.py`
  - 先做最小 smoke test 或占位测试

今天不要过早引入：

- memory
- middleware
- streaming
- structured output
- LangGraph

先把最小闭环做扎实。

## 6. 验收标准

满足以下条件，Day1 就算基本过关：

- 你能用自己的话解释 Agent 的最小工作循环
- 代码里成功定义了 3 个工具
- Agent 能至少调用其中 1 个工具
- 你能说清楚 system prompt 在这里究竟控制了什么
- 你能指出“模型推理”和“工具执行”的职责边界

## 7. 常见坑

- 把工具描述写得太模糊，导致模型不知道何时调用
- 工具返回过长文本，导致后续上下文噪声变大
- 代码一开始就引入太多能力，调试困难
- 只关心最终答案，不观察中间是否真的调用了工具
- 把 prompt 当成“文案”，而不是“行为约束”

## 8. 类比理解

可以把今天的最小 Agent 类比成一个“会思考的调度员”：

- 模型像调度员，负责判断“现在该做什么”
- 工具像不同岗位的员工，负责执行具体动作
- message 像调度台当前掌握的信息
- system prompt 像公司的管理制度

如果制度不清、信息混乱、岗位定义模糊，再聪明的调度员也会做错事。
