# Day9-问题

主题：为什么 interrupt / resume 让 LangGraph 真正具备 human-in-the-loop 能力

## 1. 常见问题

### Q1. `interrupt()` 到底做了什么？

参考作答方向：

- 暂停当前图执行
- 把当前状态通过 checkpoint 保存下来
- 向外暴露一个等待输入的 payload

### Q2. 为什么恢复时必须使用同一个 `thread_id`？

参考作答方向：

- 因为恢复的是同一条执行轨迹
- 换了 thread 就不是继续，而是新开一条执行

### Q3. 为什么 `Command(resume=...)` 很关键？

参考作答方向：

- 它不是普通参数
- 它会成为 `interrupt()` 在 node 内部的返回值
- 这就是外部输入回流到图执行内部的桥梁

## 2. 面试问题

### Q1. 为什么 interrupt 依赖 checkpoint？

作答要点：

- 因为暂停后要保存当前状态
- 否则系统无法在之后恢复执行

### Q2. 为什么说“恢复后 node 会从头执行”是一个重要语义？

作答要点：

- 因为 node 内 interrupt 前的代码会再执行一次
- 所以 interrupt 前的副作用必须谨慎，最好幂等

### Q3. approval / reject workflow 为什么是 interrupt 的经典场景？

作答要点：

- 因为流程必须等待外部决策
- 恢复值天然决定后续路径
- 非常适合展示 human-in-the-loop

## 3. 开脑问题

### Q1. 如果 interrupt 前已经执行了一个不可重复的副作用，恢复时会发生什么风险？

思考方向：

- 因为 node 会从头执行，副作用可能重复发生
- 这说明 interrupt 前的副作用必须幂等或延后

### Q2. 为什么 interrupt 比“让模型多问一句用户确认吗”更强？

思考方向：

- 因为它暂停的是系统执行，不只是生成一段文本
- 它带来的是真正的流程控制能力

### Q3. 如果系统允许用户长时间后再回来恢复，这说明底层必须具备什么能力？

思考方向：

- 持久化状态
- 可定位 thread
- 可恢复 checkpoint
