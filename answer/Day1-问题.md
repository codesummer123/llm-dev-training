# Day1-问题

主题：LangChain 最小 Agent 的本质

## 1. 常见问题

### Q1. 为什么今天不直接学 memory、middleware、LangGraph？

参考作答方向：

- 因为后续能力都建立在 model / message / tool 的最小抽象上
- 如果最小闭环没吃透，后面只会变成记 API
- 先固定系统骨架，再逐步加复杂度，调试成本最低

### Q2. 为什么工具最好返回稳定、短小、可预测的结果？

参考作答方向：

- 工具返回值会再次进入模型上下文
- 返回太长会引入噪声，干扰模型判断
- 返回结构稳定，模型更容易学会何时调用和如何使用结果

### Q3. system prompt 和 tool description 的边界是什么？

参考作答方向：

- system prompt 管全局行为边界
- tool description 管单个工具的能力边界和使用时机
- 两者都模糊时，模型最容易产生错误调用

## 2. 面试问题

### Q1. LangChain Agent 和直接调用聊天模型有什么本质差异？

作答要点：

- 直接模型调用通常是单轮或静态上下文生成
- Agent 引入了工具调用循环
- Agent 能在推理过程中访问外部能力
- Agent 的本质是“模型决策 + 工具执行 + 上下文回填”

### Q2. 为什么说 Agent 的很多问题本质上是上下文工程问题？

作答要点：

- 模型只基于当前上下文做决策
- prompt、message、tool schema 都属于上下文设计的一部分
- 工具用不好、答非所问、乱调用工具，常常不是模型智力不够，而是上下文设计差

### Q3. 你如何设计一个“最小可用”的工具型 Agent？

作答要点：

- 明确单一目标
- 限制工具数量
- 工具描述清晰
- prompt 约束明确
- 先做最小闭环，再逐步引入 memory / middleware / graph

## 3. 开脑问题

### Q1. 如果一个 Agent 有 50 个工具，能力是不是一定更强？

思考方向：

- 工具越多，上下文负担越重
- 模型的工具选择难度会上升
- “更多能力”不等于“更高可用性”
- 真实系统常常需要动态工具选择或分层代理

### Q2. 为什么我们说“模型负责推理，工具负责执行”，但现实里很多人把这两件事混在一起？

思考方向：

- 因为纯聊天体验容易掩盖系统边界
- 很多人只关注回答文本，不关注执行链路
- 把边界想清楚，才能设计可验证、可审计的系统

### Q3. 如果今天这个最小 Agent 表现不好，你会优先怀疑哪一层？

思考方向：

- prompt 是否清晰
- tool description 是否清晰
- tool 返回值是否稳定
- 消息输入是否合理
- 模型是否支持 tool calling
