# Day5-问题

主题：为什么 middleware 是 Agent 运行时控制面的关键组成部分

## 1. 常见问题

### Q1. `before_model` 和 `wrap_model_call` 的核心区别是什么？

参考作答方向：

- `before_model` 更适合做顺序化检查、状态更新、调用前拦截
- `wrap_model_call` 更适合在模型调用外层做包裹和瞬时修改
- 一个偏节点式控制，一个偏包裹式控制

### Q2. 为什么动态 system prompt 更适合做 transient 修改？

参考作答方向：

- 因为它只服务于当前这次模型调用
- 不一定值得永久写进 state
- 这类修改通常是“按当前情况临时调整”

### Q3. 为什么有些请求应该在模型调用前就被拦住？

参考作答方向：

- 因为范围太宽时，模型回答往往会空泛
- 早点收窄范围，比生成低质量长回答更好
- 这属于运行时的质量控制

## 2. 面试问题

### Q1. 为什么说 middleware 是 context engineering 的执行层？

作答要点：

- context engineering 关心“给模型什么上下文”
- middleware 让这些上下文控制在运行时真正发生
- 它把抽象原则变成可执行逻辑

### Q2. 什么场景更适合用 `wrap_model_call` 而不是 `before_model`？

作答要点：

- 当你要临时修改 system prompt、tools、model 时
- 当修改只应影响当前调用而不应持久写回 state 时
- 当你要在调用外层做控制时

### Q3. 为什么短路控制属于一种重要的 Agent 工程能力？

作答要点：

- 它能避免低质量调用
- 它能在错误输入上更早失败
- 它能帮助系统维持边界和成本控制

## 3. 开脑问题

### Q1. 如果一个系统所有逻辑都写在 prompt 里，而没有 middleware，会发生什么？

思考方向：

- 运行时控制能力变弱
- 很多逻辑难测试、难复用、难组合
- prompt 会越来越臃肿

### Q2. 为什么“动态 prompt”听起来只是小技巧，但实际上很重要？

思考方向：

- 因为它体现了系统可以按上下文实时调节行为
- 这是一种轻量但高价值的控制能力

### Q3. 如果一个请求并不错误，但范围过宽，系统应该直接拒绝、自动拆分，还是先要求用户收窄？为什么？

思考方向：

- 这取决于系统定位、风险和成本
- 先要求收窄通常更稳，但也会增加交互成本
- 自动拆分更强，但也更复杂
