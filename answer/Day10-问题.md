# Day10-问题

主题：为什么 time travel 的本质不是“回到过去”，而是“从历史上重新实验”

## 1. 常见问题

### Q1. replay 和 fork 有什么区别？

参考作答方向：

- replay 基于旧 checkpoint 重新执行后续步骤
- fork 在旧 checkpoint 上创建一个新的状态分支再继续执行
- replay 不改 state，fork 会引入新的 state 变化

### Q2. 为什么说 replay 不是“读缓存”？

参考作答方向：

- checkpoint 之后的节点会重新执行
- 所以后续逻辑、LLM 调用、interrupt 都可能再次发生

### Q3. 为什么 fork 不会覆盖原始历史？

参考作答方向：

- `update_state()` 会创建新的 checkpoint 分支
- 原执行历史仍然保留

## 2. 面试问题

### Q1. 为什么 time travel 对调试复杂图很有价值？

作答要点：

- 可以从问题点继续分析
- 不必每次从头跑
- 可以保留原轨迹再做实验分支

### Q2. 什么情况下你会优先用 replay，而不是 fork？

作答要点：

- 当我只是想从某个历史点重新执行后续步骤
- 当我不想主动修改 state
- 当我的目标是重放而不是改写路径

### Q3. `update_state()` 在 fork 中扮演什么角色？

作答要点：

- 它负责在旧 checkpoint 上创建新的状态更新
- 新的 checkpoint 成为分支起点
- 后续执行基于这个分支继续

## 3. 开脑问题

### Q1. 为什么说 time travel 和 git branch 有点像，但又不完全一样？

思考方向：

- 都是在保留原历史的同时创建新分支
- 但 LangGraph 分叉的是执行状态，不是源码文件

### Q2. 如果一个节点里有外部副作用，replay 会带来什么风险？

思考方向：

- 因为 checkpoint 之后的节点会重跑，副作用可能重复发生
- 这要求副作用具备幂等性，或者与重放逻辑隔离

### Q3. 为什么 time travel 能帮助你更好地理解“流程是怎么走成这样的”？

思考方向：

- 因为它让历史不是只可看，还可从历史上再次执行和比较

## 4. 今日输出要求

请你在完成代码后，至少自己回答下面两个问题：

1. 为什么说 replay 更像“从旧状态继续跑后续步骤”，而不是“把旧结果拿出来看看”？
- 因为 replay 不只是把旧 checkpoint 的结果读出来，而是从那个 checkpoint 开始，重新执行它后面的节点
- checkpoint 之前的步骤不会重跑，但 checkpoint 之后的步骤会再次执行
- 所以后续节点里的逻辑、模型调用、外部依赖都可能再次发生
- 这说明 replay 的本质是“从旧状态继续跑后续流程”，而不是“查看历史结果”

2. 如果 fork 后的新结果比原结果更好，这说明了什么？
- 这说明原始执行路径不一定是最优路径，而历史 checkpoint 上的某些状态值可能值得被重新审视
- 也说明上游节点生成的 state 可能还有优化空间，例如状态判断不够准确、路由依据不够好、计划生成策略不够合适
- fork 的价值就在这里：它让你可以在不破坏原始历史的前提下，验证“如果当时 state 不一样，后面会不会更好”
- 如果 fork 后结果明显更优，这通常意味着系统存在明确的优化靶点，应该回头改进产生该 state 的上游节点或决策逻辑
