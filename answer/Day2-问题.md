# Day2-问题

主题：为什么工具设计本质上是接口设计

## 1. 常见问题

### Q1. 为什么 tool description 会影响模型调用结果？

参考作答方向：

- 模型看不到你的函数实现细节
- 模型主要依据工具名、描述、参数 schema 来判断是否调用
- description 越清晰，模型越容易理解“什么时候该用这个工具”

### Q2. 为什么 schema 不只是做参数校验？

参考作答方向：

- schema 同时也是给模型看的输入说明
- 字段名、字段描述、枚举值都会影响模型如何组装参数
- 一个好 schema 不只是防错，更是在引导模型正确调用

### Q3. 为什么工具太多会让 Agent 表现变差？

参考作答方向：

- 模型的选择空间变大
- 上下文负担变重
- 相似工具之间更容易混淆
- “能力更多”不等于“可用性更高”

## 2. 面试问题

### Q1. 你如何设计一个适合 LLM 调用的工具？

作答要点：

- 能力边界清晰
- 工具名直观
- description 明确适用场景
- schema 字段名和枚举值清楚
- 返回值稳定、简洁、可继续被模型消费

### Q2. Tool schema 为什么会影响 Agent 的可靠性？

作答要点：

- schema 不清会导致参数构造错误
- schema 边界不清会导致工具误用
- schema 设计好，模型更容易稳定地产生可执行调用

### Q3. 为什么说“工具调用错误”常常不是模型能力问题，而是接口设计问题？

作答要点：

- 模型是根据你暴露出的接口信息来做决策
- 如果接口边界重叠、描述模糊、参数定义差，就容易误判
- 这是上下文设计质量的问题，不只是模型智力问题

## 3. 开脑问题

### Q1. 如果两个工具功能几乎一样，只是名字不同，会发生什么？

思考方向：

- 模型会更难稳定选择
- 结果会更依赖随机性和 prompt 细节
- 这说明“能力重复”会伤害系统可控性

### Q2. 为什么传统 API 设计经验，在 Agent 时代反而更重要了？

思考方向：

- 因为模型其实也是 API 使用者
- 只是它不是阅读源码，而是阅读 schema 和 description
- 接口设计能力会直接转化为 agent 可靠性

### Q3. 一个工具应该尽可能“万能”还是尽可能“单一职责”？

思考方向：

- 万能工具看似强大，但会让调用边界模糊
- 单一职责工具更容易让模型理解
- 真正要平衡的是“调用成本”和“边界清晰度”

## 4. 今日输出要求

请你在完成代码后，至少自己回答下面两个问题：

1. 为什么说 tool description 其实是在给模型写“使用说明书”？
- 因为模型不看函数代码和实现细节
- 模型是根据函数名、描述、参数名、参数schema来判断是否调用的，其中描述最为重要
- description越清晰，模型越能明白是否调用该函数、什么时候调用该函数

2. 如果两个工具边界重叠，你会优先改名字、改描述、改 schema，还是直接重构工具划分？为什么？
- 我会优先重构工具来划分明显边界
- 如果由于某种原因，不能重构工具，那么我会通过描述来明确区分函数的边界
