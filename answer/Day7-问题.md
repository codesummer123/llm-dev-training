# Day7-问题

主题：为什么 LangGraph 的核心不是“画图”，而是“让状态驱动流程”

## 1. 常见问题

### Q1. `State / Node / Edge` 三者分别是什么？

参考作答方向：

- `State` 是共享事实
- `Node` 是执行步骤
- `Edge` 是流转规则

### Q2. 为什么 LangGraph 不只是把 if/else 画成图？

参考作答方向：

- 因为它把共享状态、节点职责、执行流转显式建模了
- 不是为了好看，而是为了可控、可调试、可扩展

### Q3. 为什么 LangGraph 适合复杂 Agent 系统？

参考作答方向：

- 它天然适合多步骤
- 适合带状态执行
- 适合后续接 persistence、interrupt、subgraph 等能力

## 2. 面试问题

### Q1. 为什么说 state 是 LangGraph 的核心？

作答要点：

- 各节点通过同一份 state 协作
- route function 也是根据 state 做决策
- graph 的连续性本质上来自状态流动

### Q2. 条件路由和固定顺序执行的本质差别是什么？

作答要点：

- 固定顺序执行不看当前状态变化
- 条件路由会根据 state 动态选择下一步
- 这让系统更适合处理分支和非线性流程

### Q3. 为什么 node 最好单一职责？

作答要点：

- 更容易测试
- 更容易复用
- 更容易看清状态更新来源

## 3. 开脑问题

### Q1. 如果所有逻辑都塞进一个 node，会发生什么？

思考方向：

- 图会失去意义
- 调试难
- 路由不清晰
- 状态更新边界模糊

### Q2. 为什么“共享 state”既强大又危险？

思考方向：

- 强大在于各步骤能围绕同一份事实协作
- 危险在于状态设计不好就会变成混乱耦合点

### Q3. 如果一个流程未来可能加入中断、恢复、持久化，现在的 graph 设计应该提前注意什么？

思考方向：

- 节点职责清晰
- state 结构稳定
- 路由逻辑显式
- 不把所有副作用混在一个步骤里

## 4. 今日输出要求

请你在完成代码后，至少自己回答下面两个问题：

1. 为什么说 LangGraph 真正串起流程的不是节点，而是 state？
- `State` 是共享事实
- 各节点通过同一份 state 协作
- route function 也是根据 state 做决策
- graph 的连续性本质上来自状态流动
- 条件边也是根据state来决定走哪个节点
- 因为共享实时，每个节点不知道其他节点的存在，在这一点上可以说是极致解耦

2. 如果一个 route function 很复杂，你会优先改 state 设计、拆 node，还是继续堆判断？为什么？
- 我会优先检查 state 设计是否清晰，再判断是否需要拆 node，而不是继续在 route function 里堆判断
- 因为 route function 过复杂，通常说明“决策依据没有被很好地显式建模”，或者“前面的节点职责过重”
- 如果 state 含义模糊，路由判断就会越来越乱
- 如果一个节点承担了太多分析职责，也应该拆 node，让路由依据更清楚
