# Day2-训练内容

主题：LangChain Tools 不是“功能列表”，而是“给模型看的接口契约”

## 1. 学习目标

今天的目标是把 Day1 的“会挂工具”推进成“会设计工具”。

完成今天后，你应该能回答：

- 为什么有些 Agent 明明工具很多，却总是调用错
- 为什么 tool description 和参数 schema 会直接影响模型行为
- 为什么工具设计本质上是上下文工程的一部分

今天结束后，你应该具备的能力是：

- 能设计更清晰的工具边界
- 能用 `Pydantic` 定义至少一个工具参数 schema
- 能观察 Agent 到底调用了哪个工具，而不是只看最终答案

## 2. What

### 2.1 今天要学的知识点

- `@tool` 的本质
- tool description 的作用
- 参数 schema 的作用
- 为什么参数名、字段约束、字段描述会影响模型调用
- “工具太多”和“工具太模糊”分别会带来什么问题
- 如何观察工具调用轨迹
- 为什么工具设计本质上是接口设计

### 2.2 核心概念

#### Tool 是接口契约

从工程角度看，tool 是一个函数。  
但从 Agent 角度看，tool 不只是函数，而是一个暴露给模型的“能力说明书”。

模型看到的，不是你的源码，而是：

- 工具名
- 工具描述
- 参数 schema
- 参数描述
- 返回结果

也就是说，模型不是在“读代码”，而是在“读接口契约”。

#### Tool Schema 是模型的决策输入

很多人把 schema 只看成“参数校验”，这不够。

更本质的理解是：

- schema 决定模型能否理解什么时候该调用这个工具
- schema 决定模型知不知道该传什么参数
- schema 决定模型会不会把多个相似工具混淆

#### Tool Design 是 Context Engineering

工具设计不是纯后端问题，而是上下文工程问题。

因为工具描述、参数描述、返回格式，都会进入模型的决策上下文。

如果工具设计不好，常见后果是：

- 调错工具
- 参数乱填
- 调用不稳定
- 明明该调用工具却不调用

## 3. Why

### 3.1 为什么 Day2 要重点练工具设计

因为从 Day2 开始，你要建立一个更成熟的认知：

`Agent 好不好用，不只取决于模型强不强，也取决于接口是否对模型友好。`

这和传统软件工程很像：

- 面向人类开发者，API 设计差，会被误用
- 面向模型，tool 设计差，也会被误用

区别只是：

- 人类会看源码和文档
- 模型只会看你暴露给它的 schema 和描述

### 3.2 为什么很多工具调用错误，其实不是模型“笨”

常见误判是：

- “模型不聪明，所以没调对工具”

更常见的真实原因是：

- 两个工具边界重叠
- 工具描述模糊
- 参数命名不清晰
- 返回结果不稳定
- 工具太多，选择空间过大

今天要建立的认知是：

`工具调用错误，很多时候是接口设计问题，不是智力问题。`

## 4. How

### 4.1 今天的训练任务

你要实现一个“学习任务设计 Agent”，它至少有 3 个工具：

1. `lookup_topic_overview`
2. `estimate_task_effort`
3. `create_practice_task`

其中要求：

- 前两个工具可以用简单参数
- 第三个工具必须使用 `Pydantic` schema

### 4.2 代码任务拆解

#### 任务 1：设计更像“接口契约”的工具

你需要重新思考工具不是“随便包一个函数”，而是：

- 这个工具解决什么问题
- 什么时候该调用它
- 它和其他工具的边界是什么
- 它返回什么形式的结果最利于模型继续使用

#### 任务 2：实现一个 `Pydantic` 参数 schema

至少给 `create_practice_task` 设计一个 schema，建议包含：

- `topic`
- `current_level`
- `output_type`
- `minutes_available`

要求：

- `minutes_available` 做范围限制
- 枚举字段尽量明确
- 字段描述要让模型容易理解

#### 任务 3：观察工具调用轨迹

今天不要只打印最终答案。

你需要提供一种方式，能在本地观察：

- 最后有没有调用工具
- 调用了哪个工具
- 工具参数是什么

不要求做复杂日志系统，但至少要能在 Day2 结束时回答：

- 为什么这次调用选了这个工具
- 它有没有选错

#### 任务 4：比较“最终答案”和“调用过程”

你至少准备 3 类问题：

- 明显只需要解释概念的问题
- 明显需要估时的问题
- 明显需要生成练习任务的问题

每类至少跑一条，并记录：

- 最终输出
- 调用的工具
- 传入参数

### 4.3 训练建议

今天不要追求功能很多。

你更应该追求：

- 工具边界清晰
- 参数可理解
- 输出稳定
- 代码可观察

一句话：

`今天练的不是“多做几个功能”，而是“把模型能看懂的接口设计清楚”。`

## 6. 验收标准

满足以下条件，Day2 就算基本过关：

- 你能解释为什么 tool description 会影响模型选择
- 你实现了至少 1 个基于 `Pydantic` 的工具 schema
- 你能观察到至少 1 次明确的工具调用轨迹
- 你能指出 3 个工具之间的职责边界
- 你能解释“工具设计为什么属于上下文工程”

## 7. 常见坑

- 工具描述写成“介绍性文案”，而不是“使用边界说明”
- 多个工具能力重叠太多，模型难选
- schema 字段名太随意，模型不知道该填什么
- 枚举值设计模糊，比如“普通”“常规”“一般”这类边界不清的词
- 只看最终答案，不看中间调用过程

## 8. 类比理解

可以把工具设计类比成“给一个新同事写接口文档”：

- 工具名像功能标题
- description 像适用场景说明
- schema 像参数表
- 返回值像约定的输出格式

如果这些都写得很模糊，新同事会误用；模型也一样。
