## 2. 做得好的点

### 2.1 已经抓住了 retrieval 的本质

这轮最重要的不是“把 RAG 跑起来”，而是你已经开始理解：

- 模型本身没有变强
- 系统只是先把更相关的外部上下文拿进来了

这个认知非常关键，因为它决定你后面遇到 RAG 问题时，会先查检索链路，而不是盲目怪模型。

### 2.2 做出了有效的对照实验

你没有只做“有 RAG”的版本，而是把：

- 无 RAG
- 有 RAG

放在同一主程序里对照。  
这让 Day6 不只是 demo，而是一次真正有观察价值的实验。

### 2.3 检索结果可见

你把 retrieved docs 打印出来了，这一点非常好。  
因为 RAG 系统如果不看检索结果，就很难真正调试。

从工程角度讲：

- 检索到什么
- 为什么检到这些
- 这些内容有没有误导生成

往往比“最终答案看起来像不像对”更重要。

### 2.4 回答已经进入系统层思维

Day6 的两个核心回答已经比较成熟：

- retrieval 是运行时上下文补全
- 有 RAG 反而更差时，优先怀疑 retrieval 链路

这说明你已经开始从“模型使用者”转向“系统调试者”。
