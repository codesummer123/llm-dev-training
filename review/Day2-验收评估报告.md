## 2. 做得好的点

### 2.1 已经开始真正区分工具边界

这次的三个工具不再只是“多做几个函数”，而是开始体现不同职责：

- 概念解释
- 时间估算
- 练习任务生成

这种边界意识非常关键，因为后面所有 Agent 稳定性都依赖它。

### 2.2 成功接入了 `Pydantic` schema

这不是简单的“会写字段”，而是已经开始理解：

- schema 不只是校验
- schema 也是给模型看的接口说明

这一步对后续 structured output、LangGraph state schema、Deep Agents 输入设计都会产生复利。

### 2.3 开始关注调用过程，而不只看最终答案

你在 `main.py` 中实现了最小 trace 打印，这说明你已经从“只看结果对不对”转向“看系统是怎么做出这个结果的”。

这是训练中一个很重要的分水岭。

### 2.4 Day2 的问答理解是到位的

特别是这两个判断是对的：

- tool description 本质上是在给模型写使用说明书
- 工具边界重叠时，优先考虑重构边界，而不是只改文案

这说明你已经开始把问题看成“接口设计问题”，而不只是“prompt 问题”。

## 3. 踩过的坑与残留优化点

### 3.1 文件中还保留了已完成实现后的 `TODO` 注释

当前 `main.py`、`tools.py` 里还有一些最初脚手架留下来的 `TODO` 注释。  
这不会影响运行，但会影响代码可读性，容易让后续自己误判哪些功能还没做完。

建议：

- 实现完成后删除无效 `TODO`
- 只保留真正还没做的改造点

### 3.2 少数概念解释还可以更准确

例如：

- `schema` 目前的解释偏口语化
- `message` 的解释和 `system prompt` 的职责有些混淆

这不影响通过，但说明“概念表达精度”还可以继续提升。  
后面越往 LangGraph 和 Deep Agents 走，这种精度会越来越重要。

### 3.3 测试运行仍然依赖当前目录

Day2 目前和 Day1 一样，测试默认依赖在 `code/Day2` 目录下运行。  
在当前训练阶段可以接受，但后面项目变大后，建议逐步减少这种耦合。
