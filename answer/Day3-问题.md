# Day3-问题

主题：为什么结构化输出是让 Agent 可控的关键一步

## 1. 常见问题

### Q1. 为什么自由文本回答不适合直接接系统？

参考作答方向：

- 自由文本对人友好，但对程序不稳定
- 措辞一变，字符串解析就容易崩
- 程序更需要固定字段，而不是“看起来差不多”

### Q2. 为什么 schema 校验失败不是坏事？

参考作答方向：

- 它说明系统及时发现了不稳定输出
- 比起错误结果悄悄流入下游，早失败更安全
- 这是一种保护机制，而不是单纯报错

### Q3. 为什么 messages 不能只理解成聊天记录？

参考作答方向：

- messages 是模型看到的上下文
- messages 也是 agent 执行轨迹的一部分
- 工具调用、工具返回、最终回答都体现在消息流里

## 2. 面试问题

### Q1. 结构化输出为什么会提升 LLM 系统的可靠性？

作答要点：

- 输出字段固定
- 下游系统更容易消费
- 减少字符串解析脆弱性
- 失败更容易被发现和测试

### Q2. `messages` 和 `structured_response` 的角色有什么不同？

作答要点：

- `messages` 更像过程记录和上下文载体
- `structured_response` 更像最终可被系统消费的结果
- 一个偏过程，一个偏结果

### Q3. 为什么说“模型答得像对的”不等于“系统真的可用”？

作答要点：

- 文本可能语义正确但结构不稳定
- 系统依赖的不是感觉，而是可预测字段
- 工程系统需要的是可靠集成，不只是可读回答

## 3. 开脑问题

### Q1. 如果模型每次都能说出差不多的意思，为什么我们还要做 structured output？

思考方向：

- “差不多”对人够用，对程序不够用
- 系统追求的是稳定契约，不是大致正确

### Q2. 为什么很多 AI demo 看起来惊艳，但很难真正接进业务系统？

思考方向：

- 因为 demo 只关注表面效果
- 业务系统更关心边界、稳定性、失败处理和可测试性

### Q3. 如果一个需求天然就很模糊，structured output 还有意义吗？

思考方向：

- 有意义，因为它能逼你把模糊需求拆成明确字段
- 即使无法完全消除不确定性，也能把不确定性显式化

## 4. 今日输出要求

请你在完成代码后，至少自己回答下面两个问题：

1. 为什么 `structured output` 的本质不是“把回答转成 JSON”，而是“建立系统契约”？
- 把回答转成 JSON 只是外在形式，真正重要的是字段语义、字段边界和约束规则被固定下来
- 大模型可以生成自然、流畅的文本，但下游系统需要的是稳定、确定、可校验的结构化结果
- 一旦输出结构不稳定，程序就很难可靠解析，数据库写入、分支判断和自动化测试都会变脆弱
- 所以 `structured output` 的核心价值不是“换一种格式回答”，而是建立模型输出与系统消费之间的稳定契约

2. 如果一个 Agent 最终要把结果写入数据库，你会更信任自由文本还是结构化输出？为什么？
- 我会更信任结构化输出
- 因为数据库表字段本身就是确定、稳定、有类型约束的，结构化输出更符合这种系统边界
- 如果使用自由文本，程序通常还要额外做字符串解析，而这一步非常容易被措辞变化击穿
- 结构化输出不仅更容易写库，也更容易做校验、测试、失败处理和审计
